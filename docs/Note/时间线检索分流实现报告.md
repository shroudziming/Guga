# 时间线检索分流实现报告

## 背景问题

原来的记忆检索把所有 active 记忆放进同一个候选池，再对时间相关记录做加分或降权。这种方式能改善排序，但不能真正保证时间查询只在对应时间窗口内检索。因此会出现两类问题：

- 明确日期查询中，词面高度匹配但日期错误的记忆仍可能压过当天 summary。
- 用户问“我是谁”“你了解我什么”这类长期画像问题时，普通 episodic 记忆可能因为词面匹配进入 prompt，干扰画像回答。

## 解决方法

本次实现增加了一个轻量 `QueryPlan` 层，而不是引入新数据库或重写 RAG。

处理流程变为：

1. `MemoryManager.prepare_context()` 先读取 active 记忆。
2. 根据用户 query 构造 `_QueryPlan`，识别检索路线。
3. 用 `_records_for_query_plan()` 先裁剪候选池。
4. 在裁剪后的候选池内继续复用现有词法打分、语义命中融合、当前 turn 弱化和低分过滤逻辑。

支持的路线：

- `date_window`：明确日期、相对日期、上下文“那天”。
- `recent_turn`：刚才、刚刚、上一轮、previous turn。
- `last_session`：上次、上一次、上回、last chat。
- `portrait`：我是谁、你了解我、我的画像、我的偏好。
- `hybrid`：默认语义 + 词法混合检索。

## 为什么有效

关键变化是“时间/会话意图先裁剪候选，再排序”，而不是“先全局排序再给时间相关项加分”。

对于明确日期：

- 只有 `_record_matches_day()` 命中的记录进入后续排序。
- 日期匹配优先使用语义时间 `semantic_day/valid_at`。
- 如果没有语义时间，则对旧数据兼容使用 event summary 的 `day` 或记录 `created_at`。

对于“上次”：

- 先找最近一个非当前 session。
- 候选池只保留该 session 的记录。
- 旧 session 不再因为词面相似进入 prompt。

对于“刚才”：

- 候选池只保留当前 session 记录。
- 同时继续加载最近 raw session messages，让“刚才你说什么”能优先命中最近 assistant/user turn。

对于画像：

- 候选池直接置空。
- 回答依赖 `profile.json` 中的 `portrait_summary`，避免普通 episodic 噪声干扰。

## 稳定性兼容

相对时间表达如“下周三”存在一个特殊问题：旧数据可能只存了“下周三”这几个字，没有当时解析出的 `valid_at`。如果此时强制按当前日期窗口过滤，会误伤旧记忆。

因此实现采用保守规则：

- 明确日期和“那天”严格按日期窗口过滤。
- 相对日期如果能命中历史候选，则按窗口过滤。
- 相对日期如果只命中当前 turn 或没有历史候选，则回退到原有 hybrid 检索。

这个策略保证新时间字段能发挥作用，同时不破坏旧 MemoryBank 风格记录。

## Debug 可观测性

`retrieve_start` 日志现在会输出 `query_plan`，包括：

- `route`
- `reason`
- `day`
- `preferred_session_id`

这样后续看 debug report 时，可以直接判断检索错误来自“query 分类错了”，还是“候选池内排序错了”。

## 后续建议

这次只是把“检索分流”落到现有 JSONL + RAG 架构里。下一步如果继续增强时间线，应增加真正的 interval 查询能力：

- `valid_from/valid_to`
- “上周/上个月/最近几天”的范围解析
- 多事件抽取
- 冲突事实失效
- debug 中输出时间窗口内候选数量
